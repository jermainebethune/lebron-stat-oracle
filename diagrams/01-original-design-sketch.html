The Stat Oracle

A natural-language question about LeBron's career goes in. Real numbers come out — because the model writes the query, never the answer.

Request flow

1
Browser

A question in plain English

"When did he score 40+ against Boston?"
↓
2
Worker · edge

Receives the request, holds the schema

The Worker is the only thing with database access. It passes the table schema to the model as context — but never lets the model touch the database directly.

↓
3
Workers AI · Llama 3.1 8B

Translates the question into SQL

One inference call. The prompt says: here is the schema, here is the question, return only a SELECT statement.

SELECT date, opponent, points FROM games WHERE opponent = 'BOS' AND points >= 40 ORDER BY date;
↓
4
Worker · validation

Inspects the SQL before it runs

Rejects anything that isn't a single read. This is the load-bearing step — see the guardrails below.

↓
5
D1 · SQLite

Runs the query, returns real rows

Every figure the user will see originates here. Nothing downstream can alter a number.

↓
6
Workers AI · second call

Writes prose about the rows it was handed

The model summarizes only what D1 returned. It is not asked to recall anything, so it has nothing to get wrong.

"Twice — 42 points in March 2012 and 41 in February 2015."

Browser

Prose answer, with the table beside it

Show the rows next to the sentence. The user can always check the model's summary against the data it summarized.

Why this architecture is the point

A chatbot asked "how many points does LeBron have?" answers from training data, and can be confidently wrong. Here the model has two jobs — write a query, then describe rows — and neither one lets it invent a statistic. Grounding generation in a real datastore is the pattern every company wants right now, and this is a small, complete demonstration of it.

Guardrails on step 4

An LLM writing SQL against your database is a genuine risk. Building the containment is the part worth talking about in an interview.

  • SELECT-only allowlist Reject any statement that isn't a single SELECT — no DROP, DELETE, UPDATE, ATTACH, or PRAGMA.
  • One statement per request Refuse anything containing a semicolon mid-query, which blocks stacked-statement injection.
  • Server-injected LIMIT Append your own row cap so a runaway query can't exhaust the response or the CPU budget.
  • Timeout and fallback If the SQL is invalid or returns nothing, say so plainly. Never let the model free-form an answer when the query fails — that's exactly where a hallucinated stat would slip in.
