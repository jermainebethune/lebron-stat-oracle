What is a Cloudflare Worker?

From the fundamental shift, to why it's cheap, to why the limits are what they are — and where the model stops working. Every example is from The Chalk Toss.

1 · The shift — you stop renting a computer

🖥️ The old way: a server

You rent a computer. It sits in ONE city.

Runs 24/7
Whether anyone visits or not — and you pay either way
Lives in one place
Someone in Tokyo waits for a round trip to Virginia
Its problems are your problems
OS patches, memory, what happens when 10,000 people arrive at once
→

⚡ The Worker: a function

You hand Cloudflare a function. They run it in ~300 cities.

Runs only when asked
Between requests it exists nowhere. Nothing to pay for, nothing to patch
Runs where the request lands
A request from Tokyo is handled in Tokyo
Scaling is not your problem
10,000 at once is 10,000 copies. You did nothing
2 · The question that unlocks it — how can they afford to run your code in 300 cities?

🧩 Not a VM. A sandbox.

A Worker is a V8 isolate — the same trick that keeps browser tabs apart. 50 tabs isn't 50 copies of Chrome; it's one process with 50 sandboxes.

 ContainerV8 isolate
Startupseconds~5 ms
Memory each100s of MB~3 MB
Per machinedozensthousands

One machine runs thousands of customers at once. Yours starts on a request and is gone after. That is why there is no cold-start problem and why idle costs nothing.

🔬 Anatomy — this is the whole app

Ordinary JavaScript. No ports, no server config, no framework.

export default {
  async fetch(request, env) {

    // who is asking?
    await authorize(request, env)

    // ask the model for SQL
    await env.AI.run(MODEL, {...})

    // run it against the database
    await env.DB.prepare(sql).all()

    return new Response(html)
  }
}

request is what the visitor sent. env holds your bindings. Return a Response and you are done.

🔌 Bindings — the good idea

Normally a database means a URL, a password, a driver, a connection pool. From 300 cities that is a nightmare.

You declare it in config
"d1_databases": [{ "binding": "DB" }]
Your code gets env.DB
No URL. No password. No pool. The platform hands your sandbox a live handle
Same for everything
env.AI · secrets injected at runtime · never in your code

This is why the repo has no credentials in it. Not discipline — architecture.

3 · So how does it produce a web page? — there is no HTML file anywhere
1
You open the URL
Worker runs, matches pathname === '/', returns ~18 KB of text with the header content-type: text/html. That header is the only thing that makes it a page — switch it to text/plain and the browser shows raw angle brackets.
The HTML is a template string in ui.js. No file on a disk.
→
2
Browser finds an image
<img src="/img/lebron-heat-ball.jpg"> is a second, separate request.
It never reaches your Worker — static assets serve it from the edge. That is why photos cost no Worker CPU.
→
3
Browser runs the <script>
This is the fork most people miss. That JavaScript does not run in the Worker — it runs on your phone.
Worker code reaches D1, AI, secrets. Browser code reaches the page. They only talk over HTTP.
→
4
You click Ask
Browser calls fetch('/api/ask'). The same Worker runs again — different branch — and returns JSON, not HTML.
The browser script writes that JSON into the page. One Worker, two kinds of request.
Why build it this way? · It can't go stale — "1,912 games" and the provenance banner are read from D1 as the page is generated · One deploy — no separate frontend host, no CORS, no build step · Generated in your city, not fetched from an origin somewhere else
4 · Every limit falls out of that one choice — they stop feeling arbitrary
⏱️ 10 ms CPU (free plan)
Thousands of isolates share a machine; one hogging the CPU would starve the rest. Waiting doesn't count — a Chalk Toss question takes 1.2s but only a few ms of CPU. The rest is waiting on AI and D1.
📁 No filesystem
There is no machine that is "yours" to write to. Anything you save goes somewhere built for it — which is why the data lives in D1.
🧠 No memory between requests
Your isolate may be discarded, or the next request may land in another city. You cannot keep a counter in a variable. This is the one that trips everybody up.
📦 3 MB of code
It has to load in milliseconds. Photos are served as static assets, not bundled into the script.
🚧 Where the model stops working — and what comes next
So far: a function, runs anywhere, remembers nothing. Perfect for The Chalk Toss, where every question is independent. It fails the moment two visitors must see the same live thing — a chat room, a live vote, a shared counter. There is no "the" Worker to hold that; there are hundreds of copies with no shared memory. Durable Objects are the answer: a named object that exists in exactly ONE place globally, single-threaded, with its own storage. Requests for room-42 from Tokyo and London reach the same object.

A Worker is a function Cloudflare runs everywhere, instead of a server you run somewhere · examples from chalk.jermainebethune.com
