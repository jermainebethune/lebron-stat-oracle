The Chalk Toss

A natural-language question becomes a SQL query, the query returns real rows, and a second model describes only those rows. The model is never asked what it knows.

Request path

Dashed branches are the five ways the system declines to answer. They matter more than the happy path — each one is a place where a lesser design would return a confident number instead.

1
Browser → two entry points

A question in plain English

chalk.jermainebethune.com serves people. *.workers.dev serves API clients — the zone's bot protection challenges non-browser traffic on the custom domain, so both stay live on purpose.

2
Worker · src/access.js

Validates input, then checks who is asking

Rejects empty or oversized questions, then requires either a Turnstile token (what the page sends) or an x-api-key header (for programmatic use). Also reads data_provenance — if the dataset were ever unverified, every response says so.

Refusal 0 · not authorizedNo Turnstile token and no valid key. This check runs before any inference — it is the line where spending starts, and it was added after realizing anyone could drain the daily Neuron budget with a loop.
3
AI Gateway → Workers AI · qwen2.5-coder-32b

Writes a SQL query

Given the schema and the question, temperature 0. Asked for a statement — never for an answer.

SELECT date, opponent, points FROM games WHERE opponent = 'BOS' AND points >= 40
Refusal 1 · out of scopeAwards, salary, "who's the GOAT" — the model returns UNANSWERABLE and no query runs.
4
Guard · src/guard.js

Validates the SQL before it runs

An allowlist with a preprocessing pass, not a keyword blocklist. Strips comments and string literals first so nothing can hide a second statement, asserts a single SELECT, then scans for forbidden constructs and caps the row limit. 15 unit tests.

Refusal 2 · rejected queryStacked statements, mutations, comment smuggling — rejected with a reason. The model never gets a second try at improvising an answer.
5
D1 · SQLite at the edge

Runs the query, returns real rows

22 seasons of averages and all 1,912 games — minutes, points, rebounds, assists, steals, blocks, turnovers. This is the only source of any figure the user will see; nothing downstream can alter a number.

2012-06-07 | BOS | 45 2018-05-25 | BOS | 46
Refusal 3 · column doesn't existA query against data we don't have reads as "not in this database" — never as a number. An empty column would answer; a missing one cannot.
Partial · row cap hitThe guard injects LIMIT 100. When a query has more results than that, the answer says "showing the first 100 — there are more" rather than implying it is complete. Found by the eval harness on its first full run: "40+ games" has 108 results and was being presented as all of them.
Refusal 4 · zero rowsDecided in code, before any model call. This branch was once a prompt instruction and the model fired it wrongly 1 run in 3. It also removes 22% of inference calls.
6
AI Gateway → Workers AI · llama-3.2-3b

Describes the rows it was handed

Only ever sees a non-empty result set. Not asked to recall anything, so it has nothing to get wrong — and a bigger model couldn't improve a summary whose facts are already fixed.

7
Browser

Answer, query, and source rows together

The prose never appears alone. The SQL that ran and the rows it returned sit directly beneath it, so any reader can check the sentence against the data it came from. That transparency is the product.

The two model calls

Different jobs, different models, measured cost. The split is the whole cost story.

Step 3 · writes SQL @cf/qwen/qwen2.5-coder-32b-instruct 33.2 Neurons per call — 94% of total spend. Code-specialized because a wrong query is the biggest source of wrong answers.
Step 6 · writes prose @cf/meta/llama-3.2-3b-instruct 2.3 Neurons per call. Small on purpose — the facts are already fixed by the time it runs. A repeat question costs 0, served from the gateway cache in ~0.4s.

Verifying it still works

Every bug listed on this page was a SQL-generation error found by hand, one at a time. npm run eval is the systematic version — 20 questions, each paired with a query we wrote and trust.

It found the row-cap bug above on its first run, and settled the model question with a measurement: qwen2.5-coder-32b scores 20/20 at 33.2 Neurons a call; llama-3.2-3b scores 14/20 at 2.3. The cheap model failed three refusals — answering "how many championships?" with SELECT COUNT(opponent) FROM games WHERE playoff = 1. 14× cheaper was not worth 30% wrong.

Grounding a model in a datastore prevents invented numbers. It does not prevent a wrong query from producing a confidently wrong answer.

"How many MVPs did he win?" once returned 0 — the awards column existed but was empty, so the query succeeded and produced a real, meaningless figure. Every number was from the database. The answer was still wrong. An empty column is more dangerous than a missing one, because an empty column answers.

That is why five of the seven steps above can refuse, and why the rows are always shown next to the sentence.
